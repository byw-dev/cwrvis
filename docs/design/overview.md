# 系统总体设计 — cwrvis

> 文档版本：v1.0  
> 阶段：第一阶段 MVP

---

## 系统定位

cwrvis 是一套气候/水资源格点数据可视化与区域统计分析系统，采用 B/S 架构。第一阶段目标为快速交付可演示版本，满足甲方验收要求。

系统的核心设计原则是**预计算优先**：将所有计算密集型任务（netcdf 解析、区域统计、报告生成）移到离线脚本阶段完成，运行时只做数据读取和展示，以最大限度降低后端复杂度和服务器资源要求。

## 数据流

### 格点可视化数据流

```
用户选择 var + 时间 + 颗粒度
    │
    ▼
前端拼接 URL：/grid/{gran}/{var}.json
    │
    ▼
fetch JSON（浏览器缓存友好）
    │
    ▼
传入 Web Worker
    │
    ▼
双线性插值 → 色卡映射 → 阈值过滤 → ImageBitmap
    │
    ▼
MapLibre ImageSource 更新 → 渲染到地图
```

### 区域统计数据流

```
用户点击预设区域
    │
    ▼
GET /api/v1/stats?region_id=X&granularity=year&year_start=2000&year_end=2024&var=precipitation
    │
    ▼
FastAPI 查询 SQLite → 返回时间序列 JSON
    │
    ▼
ECharts 渲染折线图 / 柱状图
```

### 报告下载数据流

```
用户选择区域 + 颗粒度 + 时间范围 → 点击下载
    │
    ▼
GET /api/v1/report/download?region_id=X&granularity=year&start=2000&end=2024
    │
    ▼
FastAPI 按命名规则拼接文件路径 → 检查文件存在 → 返回文件流
（或 302 重定向到 Nginx 静态路径）
```

## 关键约束与决策记录

### ADR-001：格点数据以 JSON 静态文件分发，不提供实时 API

**决策**：格点数据预生成为独立 JSON 文件，由 FastAPI StaticFiles 直接托管，前端直接 fetch。

**原因**：
- 消除运行时 netcdf 解析，零计算压力
- 浏览器 HTTP 缓存天然生效，重复访问同一帧无网络开销
- 部署与运维极简，无需专门的数据服务

**代价**：
- 预生成脚本运行一次需要较长时间（可接受，离线完成）
- 磁盘占用约 800MB（可接受）
- 数据更新需要重新生成（第一阶段数据不更新，可接受）

### ADR-002：区域统计使用 SQLite，不使用 PostgreSQL

**决策**：使用 SQLite 存储区域统计数据。

**原因**：
- 数据量约 16 万行，SQLite 完全胜任
- 无需独立数据库服务，部署简单
- 只读访问为主，无并发写入问题

### ADR-003：报告文件预生成，不动态生成

**决策**：所有报告 `.docx` 文件由同事离线生成，系统只负责按命名规则提供下载。

**原因**：
- 完全消除后端 Word 文档生成的复杂性
- 动态生成报告需要图表渲染，实现成本高
- 固定组合的报告数量有限，预生成可行

### ADR-004：高德地图作为底图，接受 GCJ-02 偏移

**决策**：使用高德栅格瓦片作为地图底图，格点渲染采用整体边界框修正而非逐点坐标转换。

**原因**：
- 1度网格自身精度约 100km，GCJ-02 偏移约 500m，视觉上不可见
- 逐点转换性能差，且收益为零
- 高德底图与从高德平台下载的区域 GeoJSON 坐标系天然一致
